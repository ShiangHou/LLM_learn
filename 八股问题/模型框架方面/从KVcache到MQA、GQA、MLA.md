## KV cache

首先先说好，KV cache是在推理的时候用的，训练的时候不用

所以我们需要先知道推理的时候会发生什么

假如我向模型输入了两个字“我爱”，模型往下去预测“吃苹果”，

当然这里做了简化，假设“我”就是一个Toekn吧，即一个字是一个token，实际上并非如此简单，具体的Tokenizer去看tokenizer的部分

当第一个“我”字输入的时候会有一个q_我，k_我，v_我，然后一个全连接层，

然后Q K V是拼接的矩阵，这里只有一个，所以就是1

然后下一个爱，此时也一样发给大模型，现在里面是

计算 Q, K, V：经过线性层，算出了这 2 个词的特征。
$Q_{\text{我,爱}}$ 形状：(2, 4)  (2个词，假设每个词4维)

$K_{\text{我,爱}}$ 形状：(2, 4)

$V_{\text{我,爱}}$ 形状：(2, 4)

我们用Q与K与V计算巴拉巴拉往后推理，模型输出“吃”这个字

然后

1:如果没有用KV cache

当我要预测下一个字，我必须要输入

当预测“苹”的时候，我们的输入必须是完整的 “我 爱 吃” 3 个词。那么
$Q$ 的形状是 (3, 4)，

$K$ 的形状也是 (3, 4)。

矩阵乘法是 (3, 4) @ (4, 3) = (3, 3)。

在这个 (3, 3) 的矩阵里：第 1 行是“我”对所有词的注意力。第 2 行是“爱”对所有词的注意力。第 3 行是“吃”对所有词的注意力。


但是！因为因果掩码的存在，第 1 行和第 2 行的结果，在上一次预测时就已经算过了，而且绝对不会发生改变！
我们真正需要的，仅仅只是这 3x3 矩阵的最后 1 行（也就是“吃”对所有词的注意力）

那么情况其实可以，首先我们先把“我爱”的K和V先存下来

新来一个后，依旧是计算“吃”的q k v（注意这里是小写，即只计算“吃”这一个字的），然后把那个k直接拼接在大K里里面，v也是，然后

系统把新算出来的
$k_{\text{吃}}$ 和 $v_{\text{吃}}$，
追加拼接（Concat）到刚才存好的 Cache 后面：
 
更新后的 K Cache：[K_我, K_爱, k_吃]，形状变成了 (3, 4)
 
更新后的 V Cache：[V_我, V_爱, v_吃]，形状变成了 (3, 4)

那么，我现在直接用这个q吃这个和K cache去计算，这样就得到了吃和“我爱吃”的注意力得分

然后再去
1. 计算注意力分数（Scores）用当前词的 $q$ （形状 1x4），去乘以整个 K Cache 的转置 （形状 4x3）：$$Scores = q_{\text{吃}} \times K_{cache}^T$$形状推导：(1, 4) @ (4, 3) = (1, 3)

这 (1, 3) 的结果是什么意思？它完美地代表了 “吃” 这个词，对 “我”、对 “爱”、对 “吃”自己 的 3 个相关性打分！（比如算出来可能是 [0.2, 0.7, 0.1]，说明“吃”和前面的“爱”关系最紧密）

2. 聚合历史特征（Values）对这 (1, 3) 的分数做 Softmax 后，再去乘以整个 V Cache （形状 3x4）：$$Output = Scores \times V_{cache}$$形状推导：(1, 3) @ (3, 4) = (1, 4)

最终，输出的这个 (1, 4) 的向量，就是融合了“我爱吃”全部上下文语义的“吃”的特征向量。模型拿这个向量继续往后走，最终预测出了 “苹”

经典问题，为什么只有kv cache，没有什么qv cache

原因在于，q是不会复用的，在推理的时候，一个词一个词的来，q是不断变化的，而k和v是一次计算，然后后面都可以去读取他


那么这个相当于是时间换空间，用On的复杂度去换ON2的复杂度，

## MQA
聪明的你肯定想到了，如果我的序列very的长，那么我存在内存的KV矩阵不就是very的大嘛，直接猛猛占用现存，

具体占用多少现存，我们一起推理一下

首先FP32 ，即一个参数占用32个bit，四八三十二，也就是4个byte，还有半精度 16bit，

假设我们的长度是L，这里依旧是入进来的Token数量

然后维度是d_k吧，那么其实存了$L \times d_k$个数字，MHA，假设有$H_{kv}$个头，那么kv cache存了$L \times H_{kv} \times d_k$个数字，此外，K存一个，V存一个，假设我们用的是半精度，即每个数字占用2个byte，那么最后是

$$\text{Memory} = 2 \times L \times H_{kv} \times d_k \times 2 \text{ Bytes}$$

其中$d_k = d_{model} / H_q$即单个头的维度

在MHA中，q的头数应该和k和v的头数是一样的，

随着自回归解码的进行，每次生成一个新 Token（即 $L \rightarrow L+1$），系统需要向显存中追加大小为 $2 \times H_q \times d_k \times 2$ Bytes 的特征数据。当 $H_q$ 较大（例如 32 或 64）时，随序列增长而累积的显存基数极其庞大（因为此时q头数和k和v的头数是一样的）


MQA说，因为每次我的q都要和K和V乘，

举例子

在标准的 MHA 中，由于 $H_{kv} = H_q = 2$，KV Cache 必须为过去的词（例如“爱”）严格分配 2 个对应的 Key 头。内存中的历史 K 缓存（针对“爱”这个词）：$K^{(1)}$ (第一个头)：[0.3,  0.4]$K^{(2)}$ (第二个头)：[0.5,  0.6](注意：为了存下这两个头，显存消耗了 4 个浮点数空间)严格对齐的矩阵乘法：$Q^{(1)}$ 必须且只能与 $K^{(1)}$ 计算点积：[0.1, 0.2] $\cdot$ [0.3, 0.4] = 分数 A$Q^{(2)}$ 必须且只能与 $K^{(2)}$ 计算点积：[0.8, 0.9] $\cdot$ [0.5, 0.6] = 分数 B


MQA说，我保持q的头不动，k和v只要一个头不就行了？

$K^{(shared)}$ (唯一共享头)：[0.5,  0.5]
(注意：此时显存仅消耗了 2 个浮点数空间，体积缩小了一半)

现在我们有 2 个 $Q$ 头，却只有 1 个 $K$ 头，底层算子会触发广播（Broadcasting），让这两个 $Q$ 头去读取同一个物理内存地址的数据：$Q^{(1)}$ 与共享的 $K^{(shared)}$ 计算点积：[0.1, 0.2] $\cdot$ [0.5, 0.5] = 分数 C$Q^{(2)}$ 也与同一个 $K^{(shared)}$ 计算点积：[0.8, 0.9] $\cdot$ [0.5, 0.5] = 分数 D


聪明的你肯定想到了，在之前的MHA的时候，我们是在隐藏维度上直接切分，然后做的，这里你可以发现，q如果是4维，那么切成2个头后就剩下了2维，但是k和v没切啊，怎么也变成了2维呢

直接斩杀砍掉，即

```python 
# MQA 直接在源头就改变了权重矩阵的形状！
self.W_q = nn.Linear(d_model, d_model) # 4 -> 4 (依然算4，然后劈成两个2的Q头)
self.W_k = nn.Linear(d_model, d_k)     # 4 -> 2 !! (直接砍半，生出来就是一个只有2维的唯一K头)
self.W_v = nn.Linear(d_model, d_k)     # 4 -> 2 !!
```

从投影矩阵那里就入手就行了

所以KV cache相当于直接减少到了原来的1/h（原来有h个，现在只有一个），并且，由于W_q也小了，所以参数也少了

苏神说：
此外，注意到MQA由于共享了K、V，将会导致Attention的参数量减少了将近一半，而为了模型总参数量的不变，通常会相应地增大FFN/GLU的规模，这也能弥补一部分效果损失

## GQA

对于MQA，所有的 $Q$ 头（代表了模型对句子不同侧面的理解，比如有的 $Q$ 负责看语法，有的 $Q$ 负责看情感）都被迫去和一个一模一样的 $K$ 计算分数。这导致模型丧失了细粒度的特征分辨能力，在做复杂逻辑推理或写代码时，性能会明显下降

那么聪明的你肯定想到了，一个头太少，满头太多，我直接折中不就行了，

假设我们有 32 个 Query 头（$H_q = 32$）：

MHA：配置 32 个 KV 头。（比例 1:1）

MQA：配置 1 个 KV 头。（比例 32:1）

GQA（以 8 组为例）：配置 8 个 KV 头。那么每 4 个 Query 头，就被划分为一个“小组（Group）”，它们在这个小组内部共享 1 个 KV 头。（比例 4:1）

假设 $d_{model} = 4096$，每个头 $d_k = 128$

那么W_k和W_v就nn.Linear(4096, 8 * 128)

## MLA

低秩联合压缩（Low-Rank Joint Compression）。这就是 MLA 中 “Latent（潜在的、隐藏的）” 的核心含义。

实际上，苏神说，GQA其实也是一种投影


我们先看MLA是怎么做的吧

首先，假设模型总维度 d_model = 4096，Q 的头数 H_q = 32，单头维度 d_k = 128。

首先，我先用一个，假设是512维度的矩阵，叫ckv

思想是，我用这个ckv作为压缩，即，每次用的时候，直接用这个ckv去还原出整个的4096维度，然后再去和q相乘就行了

那么之前的wk和wv也变成了这个解压矩阵，即512,*4096（升维就转置就行了，）

在工程上其实是这样的，

注意到

$$Scores = Q \times K^T$$

但是现在没有K了哈哈哈，只有$$K = c^{KV} \times W_{UK}$$

所以是

$$Scores = Q \times (c^{KV} \times W_{UK})^T$$

拆开后是

$$Scores = Q \times W_{UK}^T \times (c^{KV})^T$$

由于矩阵乘法满足结合律，我们可以优先计算前面两项！
让原始的 $Q$ 直接乘以 $K$ 的解压矩阵转置，定义为一个魔改版的查询向量 $Q'$

$$Q' = Q \times W_{UK}^T$$

$$Scores = Q' \times (c^{KV})^T$$

 $Q$ 成功把 $K$ 的解压操作吸到了自己体内。在计算分数时，魔改版的 $Q'$ 直接去和显存里那个干瘪的压缩包 $c^{KV}$ 算内积，全程没有生成过全量的 $K$！

 然后第二步

 $$\text{输出} = Scores \times V \times W_O$$

 同样也没有V了

 $$V = c^{KV} \times W_{UV}$$


 $$\text{输出} = Scores \times (c^{KV} \times W_{UV}) \times W_O$$

 最后就是

 $$\text{输出} = Scores \times c^{KV} \times W_O'$$

 $$\text{最终输出} = \text{Softmax}(Q' \times (c^{KV})^T) \times c^{KV} \times W_O'$$

### 训练和推理的参数差异

骚猪，我想你一定问了，既然融合了，那么是不是只需要Wq，一个Wo，一个WUK

不是！首先一个错误就是，除了一个升维，还有一个降矩阵，

训练的时候有

降维压缩矩阵：$W_{kv\_down}$：用于将输入向量 $h_t$ 映射为低维的共享潜在向量 $c^{KV}$。

内容特征生成与解压矩阵：

$W_Q$：常规的 Query 投影矩阵（仅负责内容语义）。

$W_{UK}$：Key 的升维解压矩阵（将 $c^{KV}$ 映射为多头 K）。

$W_{UV}$：Value 的升维解压矩阵（将 $c^{KV}$ 映射为多头 V）。

解耦的位置编码（RoPE）矩阵：$W_{q\_rope}$：用于生成 Query 中专门携带位置信息的向量。

$W_{k\_rope}$：用于生成 Key 中专门携带位置信息的向量。

输出投影矩阵：$W_O$：用于将多头注意力提取后的特征映射回模型的原始维度。

但是在推理的时候是

在最终的推理代码中，实际驻留在显存并参与前向传播的参数矩阵被精简为 5 个：

核心计算融合矩阵（矩阵吸收的结果）：

$W_Q'$：由 $W_Q$ 与 $W_{UK}^T$ 融合而成。负责直接将 $h_t$ 映射为能够与 $c^{KV}$ 计算内积的查询特征。

$W_O'$：由 $W_{UV}$ 与 $W_O$ 融合而成。负责将加权后的 $c^{KV}$ 特征直接映射为注意力模块的最终输出。
(注：此时 $W_Q, W_{UK}, W_{UV}, W_O$ 这四个原始矩阵在物理显存中已被彻底销毁。)

状态保留矩阵（必须独立存在的运算）：

$W_{kv\_down}$：必须保留，因为每个新输入的 Token 仍需要通过它计算出 $c^{KV}$ 并写入 KV Cache。

$W_{q\_rope}$ 与 $W_{k\_rope}$：必须保留。因为旋转位置编码（RoPE）包含非线性的三角函数运算，数学上严禁与其他线性矩阵发生融合，必须单独计算并附加在最终的 $Q'$ 和 $K$ 上。

### 位置编码

因为位置编码是动态的，不能这样去弄，所以可以把位置编码单独的弄出来，

也就是说，弄一个W_{q\_rope}和W_{k\_rope}，这俩作为一个独立的去存储起来

$K$ 端（缓存端）的处理当输入的 $h_t$ 进来准备生成缓存时：

第一步：生成内容压缩包$h_t \times W_{kv\_down} \rightarrow c^{KV}$ (512维)这是毫无位置信息的纯内容。

第二步：生成独立位置向量 $k_r$$h_t \times W_{k\_rope} \rightarrow k_p$ (未旋转的 64 维特征)对其施加 RoPE 旋转！ $\rightarrow k_r$ (旋转后的 64 维位置向量)

第三步：塞进 KV Cache物理显存里，最终塞进去了两样东西，像两节车厢一样挂在一起：[ c_KV (512维), k_r (64维) ]。每个 Token 实际占用的缓存大小是 576 维

V也是这样

$Q$ 端（查询端）的处理当新词进来准备做查询时，它的 Query 同样被劈成了两半：第一步：生成内容查询 $Q_c$（并执行矩阵吸收）生成 4096 维的原始 $Q$。原始 $Q$ 直接乘以那个魔改的、吸收了解压矩阵的 $W_{UK}^T$，变成能够和 $c^{KV}$ 直接内积的 $Q_c$。第二步：生成独立位置查询 $q_r$输入 $h_t \times W_{q\_rope} \rightarrow q_p$ (64 维)对其施加 RoPE 旋转！ $\rightarrow q_r$ (旋转后的 64 维位置向量)此时，最终去执行进攻的 Query 队伍，也是拼接在一起的：[ Q_c, q_r ]

最后是

现在我们的 $Q$ 是 [ Q_c, q_r ]。我们的 $K$ 缓存是 [ c_KV, k_r ]。根据初中数学的向量点积分配律：$(A, B) \cdot (C, D) = (A \cdot C) + (B \cdot D)$$$\text{最终注意力分数} = \underbrace{(Q_c \cdot c^{KV})}_{\text{纯粹的语义关联度}} + \underbrace{(q_r \cdot k_r)}_{\text{纯粹的相对位置关系}}$$
