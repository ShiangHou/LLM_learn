# Tokenization
## 什么是Tokenization(分词)

大语言模型的本质在于，**在一个有限的整数序列上去计算概率**，那么首先要做的第一步就是，怎么样去**表达这个序列**，即如何把一些字符串去转化成序列，转化成什么序列。**Tokenizer**就应运而生了。

**Tokenizer的作用**一般有两个，一个可以叫做是Encoder(编码)，即把现实世界的可读取的字符串转化为序列,一个可以叫做decoder(解码),即把序列转化为字符串

###Unicode基础知识
#### 计算机的基础存储单位

想了解这些，需要一点计算机底层知识。只有从底层构建开始讲起，建立系统的知识树，才能更好的理解全貌

众所周知，计算机是由0和1构成的，那么这里的最小单位就称之为**bit**,即要么是0，要么是1。所以说，1bit最多可以表达$2^1$种。（比如单个的灯泡）

但是人们发现，1个bit存不了什么像样的东西啊，每次都读写1bit效率非常之低下，因此为了方便起见，人们就把**8个bit捆绑在一起,组成一个基础的数据块，就叫做byte(字节)**,所以一个byte可以表达$2^8 = 256$个状态（从取灯泡变成取一排8个灯泡）

#### 如何表达各种字符串
就像汉语字典、英文字典一样，计算机需要读懂现实世界的所有的数字、英文、中文、emoj表情，这需要一个映射表（即所谓的字典）。

就像我们刚才所说的encoder一样，Unicode本质上也是一种将字符串编码为序列的方式（只不过大模型不用这种，后面再说原因），经过Unicode后，每一个字符都会转化为一个唯一且永远不变的整数编号，称之为Code Point（码位），通常写成 U+ 加上十六进制数字

**什么是 Code Point（码位）？**
 
简单来说，码位就是 Unicode 分配给特定字符的纯粹的数学整数

字母 A 排在第 65 号。

汉字 你 排在第 20320 号。

表情 🌍 排在第 127757 号。

注意：它只是一个概念上的序号，它并不关心计算机的内存或硬盘到底用几个字节来存它（那是 UTF-8 干的活，后面会讲）。

**U+4F60 到底是什么意思？**

计算机科学里，为了方便表示和统一规范，大家不会直接写十进制的“第 20320 号”，而是用一种标准格式来书写码位，这就是 U+4F60。我们可以把它拆开来看：

U+：这是 Unicode 的标准前缀标识符。

4F60：这是一个**十六进制（Hexadecimal）**的数字。计算机底层喜欢逢 16 进 1（用 0-9 加上 A-F 来表示）。如果你把十六进制的 4F60 换算成我们人类熟悉的十进制：$4 \times 16^3 + 15 \times 16^2 + 6 \times 16^1 + 0 = 20320$。

用16进制的原因纯粹是为了和2进制契合

#### 转化为计算机底层
当介绍完bit和byte，以及Unicode编码后，引出了问题，

悬而未决的问题：Unicode 只规定了“谁是多少号”，但没有规定在计算机的硬盘或内存里，这些数字该用几个字节（Byte）来保存。

这就引出了 UTF（Unicode Transformation Format，Unicode 转换格式）家族。它们负责解决**如何把这些编号变成计算机认识的 0 和 1**的问题。

UTF-32：“简单粗暴”的定长方案：
核心规则：一刀切，不管你是普通的字母 A 还是复杂的表情 🌍，**统统分配 4 个字节（32个比特）**来存储。

UTF-16：“历史妥协”的变长方案
核心规则：采用 2 个字节 或 4 个字节 的变长存储。
绝大多数常用的字符（包括绝大多数汉字、常用符号）都用 2 个字节存。
那些极少用到的生僻字或后加进去的 Emoji，用 4 个字节存。

UTF-8：“统治互联网”的完美变长方案
这是目前世界上最流行、最伟大的一种编码方式，也是 Linux 服务器和当今各种大模型在底层处理文本时的绝对主力。

核心规则：采用 1 到 4 个字节 的极度灵活变长存储，根据字符的编号大小“看人下菜碟”：
1 个字节：完全等同于老旧的 ASCII 码（存所有的英文字母、数字、基础标点）。
2 个字节：存拉丁文变体、希腊字母、阿拉伯文等。
3 个字节：存绝大多数的中文汉字、日文、韩文等。
4 个字节：存 Emoji 表情、极偏僻的冷门符号。

```python 
a = "hello,你好"
print(a.encode("UTF-8"))

print(type(a.encode("UTF-8")))#会显示是byte格式

list(a.encode("UTF-8"))#这样就变成了一串整数序列，总的来说，这个编码是从0-255


```

## 使用Unicode作为Tokenizer

经过刚才的学习，我们现在明白了

Unicode是一个映射表，负责把一个字符映射到一个唯一的十六进制的符号,即从字符到**codepoint**

UTF-8 等是将Unicode映射的十六进制解码到计算机硬件上，即把codepoint解码成**byte value**，byte是8个bit，即有0-255个数字，所以说最终的byte value就是一个由0-255整数构成的数字序列。

```python 
a = "hello"
b = "你好"
list(a.encode("UTF-8"))
#结果是是 [104, 101, 108, 108, 111]
list(b.encode("UTF-8"))
#结果是 [228, 189, 160, 229, 165, 189]

```
可以看出，对于英文，一个字母就是一个token（byte），而对于汉字，一个汉字需要3个token(byte)来去存储

至此，我们已经成功解释清楚了Unicode是如何将一个字符转化为整数的，但这其实不能称得上真正的**Tokenizer**，因为Unicode只是在计算机底层做的工作，但从工具本身的角度来说，这种方法依旧是有encode和decode，还是完成了从字符串到数字序列的过程

## BPE

### 整体流程

首先说一下，我们现在依旧是在Tokenization的阶段，还没有进入embedding，现在做的全部的工作都是想着怎么把一段文字弄成一个**合理的数字化的序列**


首先，我们会拿到一系列的语料，这very的大，但现在我们就简化成

abc abc bd bc

那么首先第一步，我们要做的就是把他们全部都拆碎，拆成[a,b,c,空格，a,b,c,空格，b,d,空格，b,c]等等

我们之前学过Unicode，按照utf-8的解码方式，因此我们可以把这个单个的字符串转为0-255范围的数字，abcd分别是61，62，63，64,空格是20，那么这段文字就是

[61,62,63,20,61,62,63,20,62,64,20,62,63]

紧接着我们会统计相邻的出现的次数

62，63：3次 （即bc）
61，62 ：2次 （即ab）
63，20 ：2次（即c空格）
20，61 ：1
61，62 ：1
63，20 ：1
20 62 ：1
62，64 ：1
64，20 ：1
20，62 ：1


**注意，不要轻视空格，因为 _the 跟 the（比如出现在句首）是完全不同的**

随后开始合并，我们目前的词表是0-255，新加的话就是256，再每一次合并的时候，是合并频率最高的，并且删除对应的子对的数量

第一次合并，合并62，63，即我们的字符串变成[a,bc,空格，a,bc,空格，b,d,空格，bc]，我们把bc定义成256，

然后更新词表，在更新词表的时候需要做删除，因为刚才我们的词表里面有 a b c，之前是这个词是统计了
a b对的，现在bc是一体变成了256，那么之前的ab这一对就没有了，频率应该-1

目前只影响原来的bc左跟右，即abc的ab和c空格

那么新的统计频率就是
61，256 ：2次 （即a，bc）
63，20 ：0次（即c空格 2-2=0）  
20，61 ：1
61，62 ：1
63，20 ：1
20 62 ：1
62，64 ：1
64，20 ：1
20，62 ：1


那么继续，下面合并61-256 然后更新词表，循环往复

一般来说，更新训练词表的次数要均衡，比如如果少了，甚至连apple都会被拆分成 app和le，同时可以发现，合并的次数越多，词表的长度就越长，我们刚才只弄了一个，就用了256。此外，合并次数还跟我们的目标此表大小有关，目标词表大小设为 10,000，那么就需要合并10000-256次，

### Representation & Initialization(降维与初始化)

大模型无法直接处理高维且稀疏的 Unicode 字符空间（约 15 万个 Code points）。因此，BPE 的第一步是底层特征的降维。

UTF-8 字节化 (Byte Serialization)：将输入的 Unicode 语料文本 $C$ 转换为底层的 UTF-8 字节流。此时，任意文本都被降维成了一个取值范围在 $[0, 255]$ 之间的整型数组。

构建基础词表 (Base Vocabulary)：初始化词表 $V$。它包含两部分：256 个基础字节单元（Byte-level primitive units）。用户定义的特殊保留符 (Special Tokens) 集合 $S$（例如 <|endoftext|>）。此时词表大小 $|V| = 256 + |S|$。

### Pre-tokenization（边界隔离）
为了防止算法单纯拟合频率而跨越语义边界（如把句号和下一个单词的开头合并），我们需要进行预处理。

特殊符剥离 (Special Token Extraction)：从语料中精准定位并保护集合 $S$ 中的特殊符号，它们是不可变原子 (Immutable Atoms)，不参与后续的任何字节拆分与合并。

启发式分块 (Heuristic Chunking)：应用复杂的正则表达式（Regular Expressions）将剩余的字节流切分为预分词块 (Pre-tokens)。抽象结果：连续的文本序列被转换为一个多重集 (Multiset)。每个预分词块 $w$ 都可以表示为一个有序的字节元组：$w = (b_1, b_2, \dots, b_k)$。

简单理解，这一步是为了防止(你好，人真帅) 在合并的时候不要弄成(你  好人 真帅)，所以事先正则化把他们给切开

### Frequency Aggregation(聚合)

为了避免在随后的迭代中进行 $O(N)$（$N$ 为语料总长度）的全局遍历，我们需要构建两个核心的全局状态哈希表来进行空间换时间。

词频映射表 (Word Frequency Dictionary)：$D_{word}: W \rightarrow \mathbb{N}$。记录每个独立的预分词元组 $w$ 在语料中出现的总频次。

邻接对频次表 (Pair Frequency Dictionary)：$D_{pair}: V \times V \rightarrow \mathbb{N}$。遍历 $D_{word}$，提取所有相邻的字节对 $(b_i, b_{i+1})$，并以预分词的频次为权重进行累加，初始化全局的邻接组合计分板。


### Phase 4: 贪心合并迭代 (Iterative Greedy Merging)

这是 BPE 的核心算法层。设我们需要执行的合并总次数为 $K = V_{target} - 256 - |S|$。算法将进入一个严格执行 $K$ 次的 for 循环。

对于每一次迭代：

Argmax 提取 (Candidate Selection)：在 $D_{pair}$ 中寻找频次最大化的目标组合 $(t_L, t_R)$：$$(t_L, t_R) = \arg\max_{(x,y)} D_{pair}(x, y)$$

确定性打破平局 (Deterministic Tie-breaking)：如果存在多个组合拥有相同的最大频次，必须应用严格的平局打破机制以保证模型的跨平台复现性。在此作业中，规则为选择字典序 (Lexicographical order) 最大的元组。

状态提交 (State Committing)：生成新符号 $t_{new} = t_L \oplus t_R$。将 $t_{new}$ 分配新的 Token ID 并加入词表 $V$。将规则 $(t_L, t_R)$ 追加到合并规则表 (Merges List) 的末尾。

增量更新 (Incremental Graph Update)：这是优化的灵魂所在。只检索 $D_{word}$ 中包含了子序列 $(t_L, t_R)$ 的目标词 $w$。对每个目标词内部执行拓扑重组：

销毁旧边 (Edge Deletion)：将左侧旧组合 $(t_{L-1}, t_L)$ 和右侧旧组合 $(t_R, t_{R+1})$ 在 $D_{pair}$ 中的频次减去该词的词频。

节点收缩 (Node Contraction)：在 $w$ 的序列内部将 $t_L$ 和 $t_R$ 替换为单一节点 $t_{new}$。

建立新边 (Edge Creation)：将新产生的相邻组合 $(t_{L-1}, t_{new})$ 和 $(t_{new}, t_{R+1})$ 在 $D_{pair}$ 中的频次加上该词的词频。

### 产出

Output: 算法产出物 (Artifacts)当迭代 $K$ 次后，循环终止。BPE 训练正式结束，输出两个核心产物：vocab (Dict[int, bytes])：一个从整数 Token ID 映射到物理字节序列的词典。大小严格等于 $V_{target}$。merges (List[Tuple[bytes, bytes]])：一个严格按时间先后顺序排列的合并规则列表，长度等于 $K$。
